# rust-excel-streamer

Proof-of-concept Rust-to-WASM XLSX generator for the NestJS comparison sample.

## Goal

This crate validates the narrowest viable Rust path before any NestJS integration:

- compile for `wasm32-unknown-unknown`;
- expose a small Node-consumable API through `wasm-bindgen`;
- generate a valid `.xlsx` workbook with `rust_xlsxwriter`;
- confirm the output can be parsed by ExcelJS in Node.

## Current result

`rust_xlsxwriter` works for this proof of concept under Node-hosted WASM when used with its
`wasm` feature plus an in-memory finalization path (`Workbook::save_to_buffer()`).

What is explicitly not used here:

- `constant_memory`, because upstream implements it through `tempfile` and that is not assumed to
work in `wasm32-unknown-unknown`;
- filesystem-backed workbook writes;
- claims of true XLSX streaming from Rust WASM.

The current contract is final-buffer handoff: Rust/WASM materializes the finished workbook bytes,
returns a `Uint8Array` to Node, and Node can then stream that buffer onward.

## Build

Prerequisites:

- Rust toolchain installed locally
- `wasm32-unknown-unknown` target added
- `wasm-bindgen-cli` installed locally

From the sample root:

```bash
bun run build:rust-wasm
```

Generated artifacts stay local and are intentionally ignored by git:

- `rust-excel-streamer/pkg/`
- `rust-excel-streamer/target/`

## Smoke test

From the sample root:

```bash
node rust-excel-streamer/test_wasm.js
```

The smoke test validates:

- missing-artifact failure behavior;
- ZIP signature (`PK`);
- workbook validity by loading the generated bytes with ExcelJS.
